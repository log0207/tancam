REAL-WORLD IMPLEMENTATION GUIDE: ADAPTIVE TRAFFIC CONTROL (PDWP)
============================================================

This document outlines the end-to-end roadmap for deploying the Predictive Density-Weighted 
Pressure (PDWP) Traffic Management system in a real-world urban environment.

1. HARDWARE ARCHITECTURE (The Infrastructure)
---------------------------------------------
To move from simulation to reality, the junction requires "Eyes" and "Brains."

A. Edge Perception (Cameras):
   - High-Definition IP Cameras (1080p/4K) mounted on signal poles.
   - 4 Cameras per junction (one for each approach lane).
   - Infrared/Night Vision support for low-light accuracy.

B. Edge Computing Unit (The Brain):
   - Industrial PC (e.g., NVIDIA Jetson Orin or Intel NUC with GPU).
   - Installed in a weather-proof enclosure at the junction.
   - Responsible for running the YOLOv8 vehicle detection models and PDWP logic locally (Zero Latency).

C. Signal Interface:
   - Relay Controller to interface between the Industrial PC and the 230V Signal Lamps.
   - Support for standard 3-color (Red, Yellow, Green) and Countdown timers (Adaptive timing compatible).

D. Networking:
   - 4G/5G/Fiber backhaul to a Central Command Center for remote monitoring and data logging.

2. SOFTWARE STACK (The Logic)
-----------------------------
A. Computer Vision (The Sensing):
   - Model: Customized YOLOv8 trained specifically on the local vehicle mix (Autos, Two-wheelers, Tata Ace, Trucks).
   - Duty: Extracts "Lane Occupancy Score" every 200ms by detecting bounding boxes and calculating pixel density.

B. Adaptive Controller (PDWP):
   - Input: Live density scores from 4 cameras + current signal state.
   - Logic: 
     - Calculates "Pressure" (Density + Time-since-last-green).
     - Dynamically extends Green light for high-flow lanes (Predictive).
     - Cuts Green early if flow drops (Saturation Flow detection).

C. Central Dashboard:
   - Web-based monitoring tool (React/Node.js) showing real-time stats of all junctions.
   - GIS map integration showing "Green Corridors" and traffic heatmaps.

3. IMPLEMENTATION WORKFLOW (Step-by-Step)
-----------------------------------------
Phase 1: Traffic Survey & Simulation
- Data collection from existing junctions (video recording).
- Run recordings through our Simulation tool to configure PCU (Passenger Car Unit) weights for specific vehicles.

Phase 2: Edge Installation
- Mounting cameras and wiring them to the junction cabinet.
- Setting up the Edge PC and calibrate "Detection Zones" (defining where the stop-line and the 'wait-zone' are in the camera view).

Phase 3: Parallel Run (Shadow Mode)
- Run the AI logic in the background WITHOUT controlling the lights.
- Compare AI-suggested timing vs. Existing human/fixed timing.
- Validate reduction in "Empty Green" cycles.

Phase 4: Live Activation
- Connect the AI controller to the physical signal relays.
- Monitor with a field engineer on-site for 48 hours for contingency.

4. FAIL-SAFE MECHANISMS (Safety First)
--------------------------------------
A. Hardware Watchdog: If the Edge PC crashes, a physical relay immediately reverts the signal to "Fixed Timer Mode" (Backup logic).
B. Vision Failure: If a camera is obscured (fog/damage), that specific lane reverts to a default "Minimum Green" cycle to ensure fair flow.
C. Conflict Monitor: A hardware-level check that physically prevents "Green-Green" conflict (prevents accidental crossing signals).

5. ESTIMATED IMPACT & ROI
-------------------------
- Throughput: 20-25% increase in total vehicles passed per hour during off-peak.
- Average Delay: Expected reduction of 8-12 minutes per commuter daily.
- Environmental: Significant reduction in CO2 emissions from idling vehicles.
- Payback Period: Infrastructure costs often recovered within 12-18 months via fuel savings across the city population.

6. ALGORITHM LOGIC (STRICTLY CODE-BASED)
---------------------------------------
The system uses the `PredictiveAdaptiveController` class defined in `controller.py`.
It inherits from `AdaptiveController` but overrides the `update()` method for dynamic logic.

A. Pressure Calculation (`_get_lane_pressure`):
   Priority is determined by a score combining three factors:
   1. Queue Density: `scores[lane]` (Vehicle count weighted by PCU).
   2. Lane Wait Time: Sum of `(sim_time - v.wait_start_time)` for all waiting vehicles.
   3. Starvation: `self.starvation[lane]` multiplied by a factor (10.0 if > limit, else 2.0).
   
   Formula in Code:
   `Pressure = Queue_Score + (Total_Wait_Time * 0.05) + Wait_Bonus + (Starvation * Factor)`

B. Initial Green Calculation:
   - When entering a new green phase (`_enter_green`), duration is calculated based on the next lane's pressure.
   - Code: `g = BASE_GREEN + (scores[nxt] * 2.0)`
   - It is clamped between `BASE_GREEN` and `max_green_profile` (from time-of-day profile).

C. Dynamic Extension Logic (`update` loop):
   - Near the end of a green phase (less than 2.0s remaining), the code checks if `cur_score > 3.0` (Lane is busy).
   - If true, it extends the green duration by `self.extension_step` (5.0s).
   - Code: `self.green_duration += self.extension_step`

D. Saturation Flow / Early Cutoff:
   - If `cur_score < 0.3` (Lane effectively empty), a timer `_empty_since` starts.
   - If empty for > 1.0s, the code forces an early switch to the next best lane.
   - Code: `self._enter_yellow_close(nxt, "flow saturated")`

E. Critical Wait Interrupt:
   - The code scans all other lanes for a "Critical Wait" condition.
   - If any vehicle in another lane has waited > 90s (`max_other_wait > 90.0`), the system immediately terminates the current green phase.
   - Code: `self._enter_yellow_close(critical_lane, "CRITICAL WAIT")`

F. Emergency Preemption (inherited):
   - Checks for `emg_lane`. If active and green time > 8.0s, it switches immediately.
